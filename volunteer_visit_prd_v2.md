# 志工居家訪視表單電子化開發需求說明書 (PRD)

本文件完整記錄了志工居家訪視表單電子化系統的核心需求、資料庫結構、功能邏輯與部署架構，供 AI Agent 執行系統開發與程式碼生成。

---

## 一、 系統總覽與開發架構

* **前端介面：** 行動裝置優先（手機版網頁），以「大量點擊、極簡登打」為核心設計原則。
* **部署平台：** * 原始碼管理：託管於 **GitHub** 儲存庫（Repository）。
    * 網頁部署：部署於 **Render** 免費雲端平台（可採用 Node.js / Python 或靜態網頁架構）。
* **後端與資料庫：** 完全基於 **Google Sheets (Google 試算表)**。
* **資料與 API 互動：** * 後端 Google Sheets 撰寫 **Google Apps Script (GAS)** 腳本，並部署為 **Web App / API**（處理資料讀取、寫入與分隊拆分）。
    * 部署在 **Render** 的前端應用程式透過 API (Fetch/Axios) 與 GAS 後端進行跨域資料交換（需處理 CORS 權限）。

---

## 二、 Google Sheets 資料庫結構規劃 (Schema)

資料庫需包含以下 4 個主要工作表（Sheets）：

### 1. `訪視紀錄表` (總表)
存放所有志工送出的訪視原始紀錄。
* **欄位列表：**
    * `流水號` (ID)
    * `訪視日期時間` (Timestamp)
    * `主填寫人姓名` (Submitter)
    * `協同志工` (TeamMembers - 陣列/逗號分隔)
    * `所屬分隊` (Branch)
    * `案家姓名` (ClientName)
    * `案家電話` (ClientPhone)
    * `案家地址` (ClientAddress)
    * `GPS定位座標` (GPS_Coordinates)
    * `Q01`, `Q02`, `Q03`, ... (依題庫動態產生的題目代碼欄位)

### 2. `人員帳號管理`
用於前端登入驗證、小組成員選擇及後台權限控制。
* **欄位列表：**
    * `人員姓名` (Name)
    * `所屬分隊` (Branch)
    * `手機號碼` (PhoneNumber - 用於驗證碼對照)
    * `權限角色` (Role - 區分為：管理員 / 分隊承辦人 / 志工)

### 3. `分隊對照表`
管理各分隊承辦人與獨立試算表的對應關係。
* **欄位列表：**
    * `分隊名稱` (BranchName)
    * `分隊承辦人帳號` (ManagerEmail)
    * `分隊專屬試算表ID` (SubSheetID - GAS 自動同步派發的目標表)

### 4. `訪視題庫` (動態渲染源)
前端表單題目由此工作表動態抓取，方便管理者後續隨時調整。
* **欄位列表：**
    * `題目代碼` (QuestionID)
    * `題目分類` (Category)
    * `題目內容` (Content)
    * `題型` (Type - 支援：是否題 / 單選題 / 複選題 / 簡答題)
    * `選項內容` (Options - 半形逗號分隔，若為簡答則留空)

## 三、 核心功能與流程邏輯

### 1. 點擊式登入流程
* 志工開啟手機網頁，由下拉選單**選擇自己的姓名（主填寫人）**。
* 輸入個人手機號碼（或系統發送之驗證碼）。
* 前端發送請求至 GAS API，比對 `人員帳號管理` 是否相符，通過後進入表單。

### 2. 行動優先表單填寫
* **重複輸入個資：** 每一戶案家皆需重複輸入姓名、電話、地址。
* **一鍵 GPS 定位：** 地址與位置部分提供定位按鈕，利用瀏覽器 Geolocation API 自動抓取經緯度或反查地址，減少手動登打時間。
* **小組複選：** 提供一個「協同志工」的複選欄位（由同單位的名單中勾選），用以記錄共同執行任務的夥伴。
* **連線狀態：** 本系統回歸**線上填寫**（不考慮離線功能），填寫時需保持網路連線。

### 3. 題目動態渲染機制
前端必須異步讀取 GAS 提供的 `訪視題庫` 資料，並根據 `題型` 欄位動態渲染對應的 UI 元件：
* **是否題 / 單選題：** 渲染為手機畫面上大尺寸的**單選按鈕 (Radio Buttons)**，一擊即選。
* **複選題：** 渲染為大尺寸的**方格勾選 (Checkboxes)**。
* **簡答題：** 渲染為標準文字輸入框 (Textarea)。
* *備註：所有訪視題目皆設定為必答。條件式連動欄位暫不加入，留待後續擴充。*

### 4. 預覽與確認送出
* 志工填寫完所有題目後，點擊「下一步」進入**預覽畫面**。
* 預覽畫面以唯讀方式呈現所有輸入資料（含個資、定位、題目答案）。
* 若發現錯誤，可點擊「返回修改」；確認無誤則點擊「確認送出」，由前端將資料 POST 回後端 GAS API。

### 5. GAS 後端分隊自動拆分與資安權限
* **資料流向：** 志工送出表單後，資料由 GAS API 接收並寫入總試算表的 `訪視紀錄表`。
* **GAS 自動自動衍生派發：** 1. 後端 GAS 觸發程式，讀取該筆紀錄的 `所屬分隊`。
    2. 查找 `分隊對照表` 中該分隊對應的 `分隊專屬試算表ID`。
    3. 自動將該筆資料複寫至該分隊的獨立試算表中。
* **權限分級防護：**
    * **帳號管理員：** 擁有總試算表的完整權限（查閱、修正、產出全區統計報表）。
    * **分隊承辦人：** 僅被授予該「分隊專屬試算表」的檢視/編輯權限，絕對無法看到其他分隊的資料。分隊統計報表僅由各分隊承辦人自行在專屬表內產出。
    * *本系統不進行歷史趨勢追蹤。*

## 五、 Agent 執行指引 (GitHub & Render 部署架構)

1.  **專案結構設計：** * 在 GitHub 建立專案，區分前端程式碼（HTML/JS/CSS 或 React/Vue 框架）與 GAS 部署說明。
    * 配置 Render 部署所需的環境檔案（如 `package.json` 或 `index.html` 靜態設定）。
2.  **GAS API 跨域設定：** * 在 GAS 腳本中，`doGet(e)` 與 `doPost(e)` 的回傳必須使用 `ContentService.createTextOutput()` 並格式化為 JSON，以利部署在 Render 的前端跨域讀取。
3.  **前端開發：** 使用手機適配的 UI 框架（如 Tailwind CSS），確保所有按鈕適合手指點擊，並透過 AJAX 呼叫 GAS API 網址。
